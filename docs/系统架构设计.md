# 系统架构设计

> 面向专业认证的毕业要求达成度统一计算平台
>
> 技术路线：Java 17 + Spring Boot 3 + Vue 3 + MySQL 8

---

## 1. 系统总体架构

### 1.1 架构风格

采用**前后端分离的单体应用架构**（Monolithic SPA），后端提供 RESTful API，前端为 Vue 3 SPA。单体架构适合高校教务系统的典型规模（单专业百人级并发），部署简单、维护成本低，后续如有需要可平滑拆分为微服务。

```

### 3.2 核心计算引擎实现

> **实现状态：** 已于第3周完成，代码位于 `com.obe.platform.engine` 包。三个计算器均为纯静态方法，由 Service 层负责事务编排与数据持久化。

#### 3.2.1 第一级计算器 — Level1Calculator

计算逻辑：
1. 按学生分组，对每个学生计算 $C_{ij}$
2. 对每个学生，筛选映射到目标 $j$ 的考核点，分别累加实际得分和满分
3. $C_{ij} = \text{actualSum} / \text{maxSum}$
4. $\bar{C_j} = \frac{1}{N} \sum C_{ij}$

**实现要点：**
- 输入：`List<StudentScoreRecord>` + `Map<Long, AssessmentInfo>`（assessmentId → maxScore + objectiveId）
- 输出：`BigDecimal`（保留4位小数，HALF_UP 舍入）
- 空考核点保护：若某目标无关联考核点，返回 0；若某学生无有效得分（分母为0），不计入平均

#### 3.2.2 第二级计算器 — Level2Calculator

计算逻辑：$E_k = \sum_j (\bar{C_j} \times w_{jk})$，其中 $\sum_j w_{jk} = 1.0$ 对所有 $k$ 成立。

**实现要点：**
- 输入：`Map<Long, BigDecimal>`（目标达成度）+ `List<WeightRecord>`（objectiveId, indicatorId, weight）
- 内置 `WeightValidator` 归一化校验（容差 0.01），校验不通过抛出 `BizException`
- 输出：`Map<Long, BigDecimal>`（indicatorId → E_k，保留4位小数）

#### 3.2.3 第三级计算器 — Level3Calculator

计算逻辑：$G_k = \sum (E_k \times W_c)$，其中 $\sum W_c = 1.0$ 对所有 $k$ 成立。

**实现要点：**
- 输入：`Map<Long, Map<Long, BigDecimal>>`（courseId → indicatorId → E_k）+ `List<MacroWeightRecord>`（courseId, indicatorId, weight）
- 内置 `WeightValidator` 归一化校验（容差 0.01）
- 输出：`Map<Long, BigDecimal>`（indicatorId → G_k，保留4位小数）

#### 3.2.4 权重校验器 — WeightValidator

- 通用静态方法 `validateNormalization(Map<Long, BigDecimal>, String dimension)`
- 容差阈值：0.01（即 0.99 ~ 1.01 视为通过）
- 校验失败抛出 `BizException`，消息中明确指出指标点ID与当前总和

### 3.4 关键业务流程实现

#### 3.4.1 课程级计算（阶段一）事务流程

```
教师点击"计算"
    │
    ▼
[1] 校验：成绩是否已全部录入（无空值）
    │
    ▼
[2] 校验：内部贡献权重是否归一化（每个指标点 Σw = 1.0）
    │
    ▼
[3] Level1Calculator：逐目标计算 C̄_j（目标级达成度）
    │
    ▼
[4] Level2Calculator：逐指标点计算 E_k（课程级达成度）
    │
    ▼
[5] 持久化：写入 obj_achievement 和 course_achievement 表
    │
    ▼
[6] 状态变更：score_sheet.status → "LOCKED"
    │
    ▼
以上步骤在同一数据库事务中完成，任一步失败则整体回滚
```

#### 3.4.2 专业级计算（阶段二）事务流程

```
专业负责人/教务管理员点击"执行专业级计算"
    │
    ▼
[1] 前置校验：查询宏观支撑矩阵中所有支撑课程的 score_sheet 状态
    │  ┌────────────────────────────────────────────────┐
    │  │ 宏观看板数据：                                  │
    │  │ 课程A（数据结构）   score_sheet.status = LOCKED  │
    │  │ 课程B（高等数学B）  score_sheet.status = LOCKED  │
    │  │ 课程C（线性代数）   score_sheet.status = IMPORTED ← 未完成
    │  │ 课程D（大学物理B）  score_sheet.status = LOCKED  │
    │  └────────────────────────────────────────────────┘
    │
    ▼ 检测到未完成课程 → 阻止计算，返回未完成课程清单
    │
    ▼ （全部 LOCKED 时继续）
    │
[2] 校验：宏观总支撑权重是否归一化（每个指标点 ΣW_c = 1.0）
    │
    ▼
[3] 数据汇聚：从 course_achievement 表中提取各课程对各指标点的 E_k
    │
    ▼
[4] Level3Calculator：逐指标点计算 G_k = Σ(E_k × W_c)
    │
    ▼
[5] 持久化：批量写入 major_achievement 表
    │  每条记录包含：major_id, indicator_id, semester_id, achievement, calc_time, triggered_by
    │
    ▼
[6] 以上步骤在同一数据库事务（REPEATABLE_READ）中完成
```

**与阶段一的关键区别：**

| 维度 | 阶段一（课程级） | 阶段二（专业级） |
|-----|--------------|--------------|
| 触发者 | 主讲教师 | 专业负责人/教务管理员 |
| 数据范围 | 单课程单班级 | 全专业所有支撑课程 |
| 前置条件 | 成绩已导入、权重已归一化 | 所有课程已完成阶段一并锁定 |
| 事务隔离 | READ_COMMITTED | REPEATABLE_READ（避免读取过程中数据变更） |
| 计算器 | Level1 + Level2 | Level3 |
| 结果表 | obj_achievement + course_achievement | major_achievement |
| 锁定行为 | 锁定 score_sheet | 不产生新锁定（仅追加计算结果） |

#### 3.4.3 成绩解锁（勘误工单）流程

根据需求 5.3 节，成绩锁定后教师无法自行修改，需由教务管理员解锁：

```
教师发现成绩有误
    │
    ▼
教师向教务管理员提交勘误请求（线下/工单系统）
    │
    ▼
教务管理员审核通过后，调用解锁接口
    │  POST /api/admin/scores/{sheetId}/unlock
    │
    ▼
[1] 校验操作人权限（仅 ADMIN 角色可执行）
    │
    ▼
[2] DELETE 已有计算结果：清除 obj_achievement 和 course_achievement 中该班级的记录
    │
    ▼
[3] 状态回退：score_sheet.status → "IMPORTED"
    │
    ▼
[4] 教师可重新修改成绩并重新触发阶段一计算
    │
    ▼
以上步骤在同一事务中完成，确保计算结果与成绩单状态的一致性
```

### 3.5 数据溯源链路

系统核心价值在于从专业级达成度逐层下钻至学生个人考核点得分。完整溯源路径为：

```
专业级达成度 G_k
  └─ 课程级达成度 E_k × 宏观权重 W_c（macro_support_matrix）
       └─ 目标级达成度 C̄_j × 内部权重 w_jk（objective_indicator_weight）
            └─ 学生个人得分率 C_ij（student_score / assessment_point.max_score）
                 └─ 考核点得分 student_score.score
```

**溯源数据表关联：**

```
major_achievement
  ├─ major_id → sys_major
  ├─ indicator_id → indicator → grad_requirement
  └─ semester_id → sys_dict_semester
       │
       │（通过 Level3Calculator 的汇聚逻辑关联）
       ▼
course_achievement
  ├─ class_id → teaching_class → course
  └─ indicator_id → indicator
       │
       │（通过 Level2Calculator 的加权逻辑关联）
       ▼
obj_achievement
  ├─ class_id → teaching_class
  └─ objective_id → course_objective
       │
       │（通过 Level1Calculator 的求和逻辑关联）
       ▼
student_score
  ├─ sheet_id → score_sheet
  ├─ student_id → student
  └─ assessment_id → assessment_point → course_objective
```

> 此链路对应 SRS 第 6 章"数据溯源链路完整性分析"的要求，也是模块 D 中"穿透式 Excel 数据台账"的数据基础。

**关系说明：**

- `macro_support_matrix.course_id ↔ indicator_id`：该表为课程与指标点的 N:M 关联表，每一行表示某门课程支撑某个指标点，并记录对应的支撑强度（H/M/L）和总支撑权重 $W_c$
- `objective_indicator_weight.objective_id ↔ indicator_id`：该表为课程目标与指标点的 N:M 关联表，每一行表示某课程目标对某指标点的内部贡献权重 $w_{jk}$
- `class_student`：教学班级与学生的 N:M 关联表（一个学生可属于多个教学班级，一个班级包含多个学生）
- `student_score`：学生与考核点的 N:M 关联表，同时关联 score_sheet，记录每个学生在每个考核点上的实际得分

## 5. 安全与事务设计

### 5.1 认证与授权

- **认证方式：** JWT 无状态 Token，登录后签发，请求头携带 `Authorization: Bearer <token>`
- **角色模型：** 基于 `sys_role` 角色表的 RBAC，角色信息编码在 JWT payload 中，系统预置四种业务角色（系统管理员、教务管理员、专业负责人、主讲教师）
- **接口级控制：** Spring Security 注解 `@PreAuthorize("hasRole('TEACHER')")` 保护各 API 端点
- **数据级控制：** 主讲教师仅能访问自己负责的 `teaching_class` 数据，通过 Service 层过滤

### 5.2 事务隔离

- **默认隔离级别：** `READ_COMMITTED`（MySQL 默认）
- **课程级计算事务：** 使用 `@Transactional` 包裹整个阶段一计算流程，确保原子性
- **专业级计算事务：** 读取多个课程数据并写入结果，使用 `REPEATABLE_READ` 避免幻读
- **乐观锁：** `score_sheet.status` 字段配合版本检查，防止并发重复计算

### 5.3 计算锁定机制

**锁定流程（阶段一计算时）：**

```
教师提交计算请求
    │
    ▼
[SELECT ... FOR UPDATE]  -- 锁定 score_sheet 行
    │
    ▼ 校验 status = 'IMPORTED'
    │
    ▼ 执行计算 ...
    │
    ▼ UPDATE score_sheet SET status = 'LOCKED'
    │
    ▼ COMMIT
```

**解锁流程（教务管理员勘误时）：**

```
教务管理员调用解锁接口
    │
    ▼
[SELECT ... FOR UPDATE]  -- 锁定 score_sheet 行
    │
    ▼ 校验 status = 'LOCKED'
    │
    ▼ DELETE obj_achievement WHERE class_id = ?
    │
    ▼ DELETE course_achievement WHERE class_id = ?
    │
    ▼ UPDATE score_sheet SET status = 'IMPORTED', locked_at = NULL, locked_by = NULL
    │
    ▼ COMMIT
    │
    ▼ 教师可重新修改成绩并重新触发阶段一
```

> **级联影响：** 若该课程已完成阶段二（专业级计算），解锁后需提醒管理员重新执行专业级计算，因为之前的专业级结果中包含了该课程的旧数据。

└───────────────────────┬───────────────────────────┘
                        │
┌───────────────────────▼───────────────────────────┐
│                  MySQL 8.0                        │
│              Port: 3306                           │
│         Database: obe_platform                    │
└──────────────────────────────────────────────────┘
```

- **操作系统：** Linux (CentOS 7+ / Ubuntu 20.04+)
- **JDK：** 17+
- **构建：** Maven 打包后端 JAR，Vite 构建前端静态资源
- **反向代理：** Nginx 处理静态资源托管和 API 转发
- **数据库：** MySQL 8.0，配置 `innodb_buffer_pool_size` 为物理内存的 60%-70%

**docker-compose.yml：**

```yaml
version: '3.8'

services:
  obe-mysql:
    image: mysql:8.0
    container_name: obe-mysql
    restart: always
    ports:
      - "3306:3306"
    environment:
      MYSQL_ROOT_PASSWORD: ${DB_PASSWORD}
      MYSQL_DATABASE: obe_platform
      MYSQL_CHARSET: utf8mb4
      MYSQL_COLLATION: utf8mb4_unicode_ci
    volumes:
      - mysql-data:/var/lib/mysql
      - ./sql/init.sql:/docker-entrypoint-initdb.d/init.sql
    command: --character-set-server=utf8mb4 --collation-server=utf8mb4_unicode_ci
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 10s
      timeout: 5s
      retries: 5

  obe-backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    container_name: obe-backend
    restart: always
    ports:
      - "8080:8080"
    environment:
      SPRING_DATASOURCE_URL: jdbc:mysql://obe-mysql:3306/obe_platform?useSSL=false&allowPublicKeyRetrieval=true&serverTimezone=Asia/Shanghai
      SPRING_DATASOURCE_USERNAME: root
      SPRING_DATASOURCE_PASSWORD: ${DB_PASSWORD}
      JWT_SECRET: ${JWT_SECRET}
      JWT_EXPIRATION: 86400000
      SPRING_PROFILES_ACTIVE: prod
    depends_on:
      obe-mysql:
        condition: service_healthy

  obe-nginx:
    image: nginx:1.25-alpine
    container_name: obe-nginx
    restart: always
    ports:
      - "80:80"
    volumes:
      - ./frontend/dist:/usr/share/nginx/html
      - ./nginx/default.conf:/etc/nginx/conf.d/default.conf
    depends_on:
      - obe-backend

volumes:
  mysql-data:
```

**backend/Dockerfile：**

```dockerfile
# 多阶段构建：先编译，再运行
FROM maven:3.9-eclipse-temurin-17 AS build
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline -B
COPY src ./src
RUN mvn package -DskipTests -B

FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY --from=build /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

**nginx/default.conf：**

```nginx
server {
    listen 80;
    server_name _;

    # 前端静态资源
    location / {
        root   /usr/share/nginx/html;
        index  index.html;
        try_files $uri $uri/ /index.html;
    }

    # 后端 API 代理
    location /api/ {
        proxy_pass http://obe-backend:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        client_max_body_size 50m;
    }
}
```

**启动与停止：**

```bash
# 启动所有服务（后台运行）
docker compose up -d --build

# 查看运行状态
docker compose ps

# 查看后端日志
docker compose logs -f obe-backend

# 停止所有服务
docker compose down

# 停止并清除数据卷（重置数据库）
docker compose down -v
```

**两种部署方式对比：**

| 维度 | 传统部署 | Docker 部署 |
|-----|---------|------------|
| 环境准备 | 手动安装 JDK、MySQL、Nginx | 仅需安装 Docker + Docker Compose |
| 部署速度 | 较慢（逐个配置） | 一键启动 `docker compose up` |
| 环境一致性 | 依赖手动对齐版本 | 镜像固化，环境完全一致 |
| 适用场景 | 生产环境长期稳定运行 | 开发测试、快速交付、演示环境 |
| 运维复杂度 | 需熟悉各组件配置 | Docker Compose 统一管理 |
| 数据持久化 | 直接使用本地磁盘 | Docker Volume 映射到本地 |
